# 前端规范

当前 AgentFlow 仓库是 NestJS 后端项目，没有独立前端应用。本文档用于限制未来新增管理台、调试台、Web UI 或静态页面时的实现方式。

## 当前状态

- 当前没有 `frontend`、`web`、`apps/web`、`src/pages`、`src/app` 等前端目录。
- 当前只有 `src/assets/success.html` 这类静态资源。
- AI 不得假设项目已经接入 React、Vue、Next.js、Vite 或 Tailwind。

## 新增前端前的限制

新增前端代码前必须先确认：

- 用户是否明确要求新增前端。
- 前端目录放置位置，例如 `apps/web`、`web` 或 `src/public`。
- 使用的框架、构建工具和包管理方式。
- 是否需要与当前 NestJS 服务同仓构建、同进程托管或独立部署。

没有明确要求时，不要主动引入前端框架。

## UI 设计规范

如果后续新增前端页面，必须遵守：

- 默认构建真实可用的业务界面，不做纯营销落地页。
- 管理台、调试台、Agent 编排工具应采用信息密度高、安静、可扫描的界面。
- 卡片只用于重复项、弹窗或明确需要框定的工具区域，不要把页面区块层层包成卡片。
- 工具栏按钮优先使用图标，图标按钮必须有可访问名称或 tooltip。
- 表单必须有明确 label、错误提示、禁用态和加载态。
- 文本不能溢出按钮、表格、卡片或侧边栏。
- 移动端和桌面端都要检查布局，不允许元素重叠。

## 交互规范

- 涉及 API 调用的交互必须处理 loading、success、empty、error 四类状态。
- 会修改数据的操作必须提供明确反馈。
- 删除、重置、覆盖配置等高风险操作必须有确认步骤。
- 长任务、流式 Agent 输出和后台运行任务必须给出进度或持续状态。
- 不要把关键操作只藏在 hover 状态中。

## 与后端协作

- API 地址应通过配置注入，不要在组件中写死生产域名。
- 与后端约定响应格式时，要考虑当前默认成功格式：`{ code, message, data }`。
- 如果调用流式接口，要单独处理 SSE 或 chunked response，不要套用普通 JSON 请求封装。
- 表单字段必须与 DTO 保持一致，后端 DTO 变更时同步更新前端类型。

## 验证要求

新增前端后，交付前至少验证：

- 开发服务器可启动。
- 主要页面可打开。
- 核心交互无控制台错误。
- 桌面和移动端布局不重叠。
- 所有新增 API 调用都有失败态处理。
